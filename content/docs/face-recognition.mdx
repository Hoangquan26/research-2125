## Các phần trọng tâm để triển khai

| Phần | Việc cần làm | Mốc tham chiếu thực dụng |
| --- | --- | --- |
| Camera và chất lượng ảnh | Mở đúng camera, chọn frame rõ mặt, kiểm tra blur, sáng tối, kích thước mặt và số lượng mặt trong khung hình. | Chỉ xử lý tiếp khi có đúng `1` mặt, detection score ví dụ `>= 0.6`, crop mặt đủ lớn trong frame và không quá mờ/tối. |
| Pipeline local | Detect mặt, căn chỉnh, tạo embedding, so khớp cosine và quyết định theo threshold/margin. | PoC đang dùng crop `112 x 112`, embedding ONNX ~`13 MB`, `threshold 0.62`, `margin 0.08`, `topK 10`. |
| Audit và fallback | Lưu kết quả nhận diện, model, điểm số, ngưỡng, ảnh chứng cứ và chuyển sang QR/số điện thoại khi không chắc chắn. | Log tối thiểu: `best_score`, `second_score`, `threshold`, `margin`, `model_key`, `alignment_method`, `snapshot_version`, `fallback_reason`. |

## Phương án 1: Tự xử lý trong hệ thống

Phương án này đặt pipeline nhận diện trong trình duyệt hoặc trong máy chủ nội bộ do hệ thống kiểm soát. Với máy khách Gym, hướng phù hợp nhất là xử lý phần realtime ngay trên client: camera mở bằng `getUserMedia()`, client detect mặt liên tục, chọn frame đủ tốt, căn chỉnh ảnh mặt, tạo embedding và so với vector hội viên đã cache.

Luồng triển khai điển hình:

1. Dùng MediaPipe để phát hiện khuôn mặt trong luồng camera.
2. Cắt và chuẩn hóa vùng mặt.
3. Dùng model embedding như MobileFaceNet-like hoặc ArcFace mobile để tạo vector khuôn mặt.
4. So sánh embedding với danh sách hội viên đã đăng ký.
5. Tạo check-in nếu điểm tương đồng vượt ngưỡng.

### Ba lớp kỹ thuật chính

Luồng nhận diện khuôn mặt trên client nên tách thành ba lớp: **face detect**, **face alignment** và **face recognition**. Một công cụ phát hiện mặt tốt chưa đủ để nhận diện hội viên; nó chỉ tìm vùng mặt. Trước khi tạo embedding hoặc gọi API nhận diện, ảnh mặt nên được căn chỉnh để giảm sai lệch do nghiêng đầu, lệch camera hoặc crop không ổn định.

#### Face detect

Ghi chú đầu ra detector, rút gọn theo JSON mẫu:

```json
{
  "mediapipe_blazeface": {
    "boundingBox": { "originX": 120, "originY": 80, "width": 180, "height": 180 },
    "score": 0.93,
    "keypoints": [{ "name": "left_eye", "x": 165, "y": 135 }, { "name": "right_eye", "x": 225, "y": 134 }, { "name": "nose", "x": 196, "y": 170 }]
  },
  "face_api_tiny_detector": {
    "box": { "x": 120, "y": 80, "width": 180, "height": 180 },
    "score": 0.91
  }
}
```

Detector output chỉ dùng để chọn/căn chỉnh crop và quality gate; nó không phải `embedding[]` để nhận diện hội viên.

Ghi chú khác biệt: `MediaPipe Tasks` là API Google AI Edge đang dùng trong PoC để chạy `.tflite` trực tiếp từ `/public/models`. `@tensorflow-models/face-detection` là package TensorFlow.js có model option `MediaPipeFaceDetector`; nó cũng dựa trên model MediaPipe nhưng API, runtime và output schema khác. Vì vậy nên so sánh như hai hướng triển khai offline riêng.

#### Face alignment

Alignment tối ưu cho pipeline local là căn cùng một chuẩn ở cả lúc đăng ký và lúc check-in. Với model mobile ArcFace/MobileFaceNet trong PoC, cần align về crop `112 x 112`, dùng template ArcFace cố định và lưu `alignment_method` để audit.

Lý do chọn `112 x 112` không phải vì đây là kích thước “đẹp”, mà vì đó là input contract của model embedding đang dùng. Model `w600k_mbf.onnx` nhận tensor `[1, 3, 112, 112]`; nếu đưa ảnh kích thước khác thì runtime phải resize hoặc model lỗi shape. Quan trọng hơn, template ArcFace đặt mắt/mũi/miệng vào các tọa độ cố định trong khung `112 x 112`, nên enrollment và check-in phải dùng cùng crop size/template. Nếu lúc đăng ký dùng crop chuẩn nhưng lúc check-in crop lệch size, lệch padding hoặc lệch vị trí mắt/mũi, embedding sẽ đổi phân phối và `threshold/margin` đã benchmark không còn đáng tin.

#### Face recognition

Do đó các vector phải cùng hệ tham chiếu, cùng preprocessing và cùng alignment để đảm bảo độ chính xác. Nếu đổi model recognition, output embedding có thể khác chiều hoặc khác phân phối điểm, nên cần tạo lại vector hội viên và benchmark lại `threshold`/`margin`.

| Lựa chọn | Dung lượng | Input | Output vector | Tương thích vector | Tổng kết |
| --- | --- | --- | --- | --- | --- |
| MobileFaceNet-like ONNX | Tùy file model; thường nhỏ hơn ArcFace mobile nếu backbone rất nhẹ. | Thường `112 x 112` RGB, normalize theo model. | Thường `128-512` chiều; cần L2-normalize trước cosine. | Chỉ so với vector sinh bởi cùng `model_key` và cùng preprocessing. | Hợp nếu cần latency thấp; phải benchmark lại `threshold/margin` bằng ảnh Gym thật. |
| ArcFace mobile ONNX | PoC: `w600k_mbf.onnx` `13.616.099 bytes`. | PoC: `[1,3,112,112]`, RGB CHW, normalize `[-1,1]`. | PoC giả định `512` float32; mỗi vector raw khoảng `2 KB`. | Không dùng chung với descriptor face-api.js; đổi model phải re-embed toàn bộ vector. | Hướng chính hợp lý cho Gym nếu P95 check-in đạt mục tiêu `1-2 giây`. |
| face-api.js FaceRecognitionNet | Cần đo theo weight tải về; repo PoC hiện chưa bundle weight face-api.js. | Ảnh/canvas qua pipeline face-api.js; thường đi cùng detector + landmark của face-api.js. | `descriptor` của face-api.js, ngưỡng riêng, không cùng phân phối với ONNX. | Không trộn với vector ArcFace/MobileFaceNet; phải lưu `model_key` riêng. | Dùng làm baseline/demo; production chỉ sau khi đo accuracy/latency trên camera Gym thật. |

Luồng embedding và vector search, nhìn theo dữ liệu đi qua từng bước:

```txt
ảnh mặt đã align 112x112
  -> đổi ảnh thành tensor [1, 3, 112, 112]
  -> normalize pixel về [-1, 1]
  -> chạy model embedding
  -> nhận vector thô, ví dụ 512 số float
  -> L2-normalize vector để độ dài vector = 1
  -> so cosine với vector hội viên cùng model_key
  -> sort điểm giảm dần
  -> gom nhiều vector của cùng 1 member thành 1 candidate tốt nhất
  -> lấy best_score, second_score, topK candidates
  -> áp dụng threshold, margin, member status, cooldown
```

Ví dụ dễ hiểu với vector 3 chiều:

```txt
query hiện tại = [0.2, 0.4, 0.1]
vector hội viên A = [0.1, 0.5, 0.2]

cosine_score =
  0.2 * 0.1
  + 0.4 * 0.5
  + 0.1 * 0.2
  = 0.24
```

Trong thực tế vector không chỉ có 3 chiều mà thường `128-512` chiều. Nếu vector đã L2-normalize, điểm cosine càng gần `1` thì càng giống, càng gần `0` thì càng khác. Với bài toán này, PoC đang dùng ngưỡng ví dụ `threshold = 0.62`, nhưng ngưỡng thật phải benchmark bằng ảnh Gym thật.

L2-normalize nghĩa là đưa vector về cùng độ dài `1`, để điểm so sánh không bị ảnh hưởng bởi vector dài/ngắn:

```txt
norm = sqrt(v1^2 + v2^2 + ... + vn^2)
normalized_vector[i] = vector[i] / norm
```

Sau khi normalize cả query và vector hội viên, cosine similarity chính là dot product:

```txt
cosine_score(query, enrolled) =
  query[0] * enrolled[0]
  + query[1] * enrolled[1]
  + ...
  + query[n-1] * enrolled[n-1]
```

Ví dụ với Top K:

```txt
query_embedding
  -> member_a/vector_1: 0.84
  -> member_a/vector_2: 0.80
  -> member_b/vector_1: 0.61
  -> member_c/vector_1: 0.58

group theo member:
  best = member_a, score 0.84
  second = member_b, score 0.61

decision:
  0.84 >= threshold 0.62
  0.84 - 0.61 = 0.23 >= margin 0.08
  -> PASSED
```

Các tham số quyết định:

| Tham số | Ví dụ PoC | Ý nghĩa | Cách dùng |
| --- | ---: | --- | --- |
| `threshold` | `0.62` | Điểm tối thiểu để best candidate được xem là đủ giống. | Nếu `best_score < threshold` thì `FAILED`, không tự động check-in. |
| `margin` | `0.08` | Khoảng cách tối thiểu giữa best và second candidate. | Nếu `best_score - second_score < margin` thì `AMBIGUOUS`, chuyển QR/số điện thoại. |
| `topK` | `10` | Số vector/candidate giữ lại sau khi sort score. | Dùng để audit và lấy `best_score`, `second_score`; không nên chỉ log best. |
| `best_score` | `0.84` | Candidate có cosine cao nhất sau khi group theo `member_id`. | So với `threshold`; là điểm chính để quyết định passed/failed. |
| `second_score` | `0.77` | Hội viên gần đúng thứ hai. | Dùng tính `best_score - second_score`; ví dụ `0.84 - 0.77 = 0.07`, nhỏ hơn `0.08` nên ambiguous. |
| `model_key` | `buffalo_sc_w600k_mbf` | Phiên bản model tạo embedding. | Query và enrolled vector phải cùng `model_key`; khác model thì không so. |
| `embedding_size` | `512` nếu model trả 512D | Số chiều vector. | Bỏ qua vector cache nếu khác chiều với query vector. |

Ví dụ quyết định:

```txt
case_pass:
  best_score = 0.84
  second_score = 0.61
  threshold = 0.62
  margin = 0.08
  -> 0.84 >= 0.62 và 0.84 - 0.61 = 0.23 >= 0.08
  -> PASSED

case_ambiguous:
  best_score = 0.84
  second_score = 0.77
  threshold = 0.62
  margin = 0.08
  -> 0.84 >= 0.62 nhưng 0.84 - 0.77 = 0.07 < 0.08
  -> AMBIGUOUS
```

### pgvector và vector database

Nếu search chạy trên server thay vì trong IndexedDB, có thể dùng `pgvector` trong PostgreSQL hoặc một vector database chuyên dụng. Với Gym offline-first, điểm quan trọng là: server/vector DB giúp sync, enrollment, audit và benchmark tốt hơn, nhưng không thay thế được local vector cache khi máy khách mất mạng.

`pgvector` hiện hỗ trợ lưu vector ngay trong Postgres, exact nearest neighbor search, approximate nearest neighbor search, nhiều kiểu vector (`vector`, `halfvec`, `bit`, `sparsevec`) và nhiều distance operator như L2, inner product, cosine distance, L1, Hamming, Jaccard. Với cosine similarity, `pgvector` dùng cosine distance operator `<=>`; similarity có thể tính bằng `1 - distance`.

Ví dụ schema và query:

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE member_face_vectors (
  id uuid PRIMARY KEY,
  member_id uuid NOT NULL,
  model_key text NOT NULL,
  embedding vector(512) NOT NULL,
  revoked_at timestamptz
);

SELECT
  member_id,
  1 - (embedding <=> '[0.021, ...]'::vector) AS cosine_similarity
FROM member_face_vectors
WHERE model_key = 'buffalo_sc_w600k_mbf'
  AND revoked_at IS NULL
ORDER BY embedding <=> '[0.021, ...]'::vector
LIMIT 10;
```

Đánh giá vector database chuyên dụng nếu dùng cho Gym:

| Lựa chọn | Độ hỗ trợ | Khả năng truy xuất | Chi phí vận hành | Độ phức tạp hệ thống | Ưu điểm | Nhược điểm |
| --- | --- | --- | --- | --- | --- | --- |
| PostgreSQL + pgvector | Hỗ trợ trực tiếp trong Postgres qua extension; có exact search, HNSW, IVFFlat, cosine/L2/inner product và filter SQL. | Truy vấn vector + metadata trong cùng SQL: `WHERE model_key`, `revoked_at`, join `members`, order bằng `<=>`, limit Top K. | Thấp hơn nếu hệ thống đã có Postgres; thêm chi phí CPU/RAM/index trên DB chính. | Thấp đến trung bình: thêm extension, migration, index, tuning query; không cần service vector riêng. | Dữ liệu member/check-in/audit/vector ở cùng nơi, dễ transaction, dễ backup, dễ enforce trạng thái hội viên. | DB chính có thể bị tải nặng nếu search lớn; cần tune HNSW/IVFFlat và không thay thế được vector cache offline trên máy khách. |
| Vector DB chuyên dụng như Qdrant/Milvus/Weaviate | Hỗ trợ ANN/search chuyên sâu, collection, payload/filter, scale search độc lập, thường có API/SDK riêng. | Truy xuất vector nhanh ở quy mô lớn; filter metadata được nhưng phải đồng bộ payload từ DB nghiệp vụ sang vector DB. | Cao hơn: thêm service, storage, monitoring, backup, network, có thể thêm chi phí cloud/managed service. | Cao: phải đồng bộ member status/vector lifecycle, xử lý consistency, retry, rebuild index, backup hai hệ. | Tách tải search khỏi Postgres, scale tốt hơn khi vector rất lớn hoặc nhiều use case AI/search. | Tăng hạ tầng và nguy cơ lệch dữ liệu giữa vector DB và DB nghiệp vụ; audit/check-in vẫn phải lưu ở DB chính. |
| Client IndexedDB brute-force | Hỗ trợ offline thật sự trong browser, không cần server/vector DB khi mất mạng. | Truy xuất tuyến tính trên snapshot local; phù hợp Top K với tập vector theo chi nhánh. | Thấp ở server; chi phí nằm ở RAM/CPU/storage của máy khách. | Trung bình ở client: cần sync snapshot, cache model/vector, Web Worker, queue offline, conflict handling. | Máy khách vẫn nhận diện khi mất mạng; latency không phụ thuộc network; dữ liệu có thể giới hạn theo cơ sở. | Không hợp sync toàn hệ thống quá lớn; dưới khoảng `10.000` vector local là mốc dễ kiểm soát hơn, cần giới hạn snapshot theo chi nhánh. |

### Runtime chạy model trên client

Trong bài toán này, **model** là file chứa trọng số và cấu trúc mạng, ví dụ `w600k_mbf.onnx`. Model chỉ nói rằng: nhận ảnh mặt `112 x 112`, tính toán qua các lớp neural network, rồi trả ra embedding. Nhưng browser không tự hiểu file ONNX/TFLite giống như không tự hiểu một file Excel nếu không có chương trình mở file.

**Runtime** là lớp phần mềm đọc file model đó và thực sự chạy phép tính trên thiết bị. Nó nhận tensor đầu vào, cấp phát bộ nhớ, chọn CPU/GPU nếu có, chạy từng operator của model rồi trả output. Trong PoC hiện tại, ONNX Runtime Web chính là runtime; `executionProviders: ["wasm"]` nghĩa là đang yêu cầu runtime chạy model bằng WebAssembly trên CPU.

Nhìn theo luồng check-in:

```txt
ảnh mặt đã align 112x112
  -> tạo tensor [1, 3, 112, 112]
  -> đưa tensor + file model vào runtime
  -> runtime chạy model bằng WASM/WebGPU/WebGL
  -> nhận embedding 512 chiều
  -> tự code tiếp phần cosine search, threshold, margin
```

Vì vậy runtime **không quyết định người này là ai**. Runtime chỉ giúp chạy phép tính để sinh embedding. Phần quyết định match hay không vẫn nằm ở logic của hệ thống: so cosine với vector hội viên, kiểm tra `threshold`, `margin`, `model_key`, trạng thái hội viên và cooldown.

Cùng một model có thể chạy bằng nhiều runtime/backend khác nhau. Khác biệt chính không nằm ở ý nghĩa nghiệp vụ, mà nằm ở tốc độ, độ ổn định, mức hỗ trợ trình duyệt và phần cứng cần có.

| Runtime | Cách chạy | Tương thích trình duyệt | Phù hợp với model |
| --- | --- | --- | --- |
| WASM | Model được chạy bằng WebAssembly trong browser, dùng CPU; không cần GPU. | Rộng nhất: Chrome, Edge, Firefox, Safari desktop/mobile đều hỗ trợ WebAssembly cơ bản. Rủi ro chính là RAM/CPU trên thiết bị yếu hoặc CSP chặn WASM. | Phù hợp ONNX Runtime Web và model embedding như `w600k_mbf.onnx`; PoC đang dùng `executionProviders: ["wasm"]`. |
| WebGPU | Browser gửi tensor/operator xuống GPU qua WebGPU API để tăng tốc tính toán song song. | Tốt nhất trên Chrome/Edge bản mới; Safari/Firefox cần kiểm tra phiên bản và cấu hình thật trên máy khách. Không giả định có sẵn trên mọi máy. | Phù hợp khi muốn tăng tốc ONNX/TensorFlow.js trên GPU; chỉ bật sau khi benchmark cùng model vì score quanh threshold có thể lệch nhẹ. |
| WebGL | Dùng GPU graphics pipeline; tensor thường được biểu diễn qua texture/shader WebGL. | Rộng trên Chrome, Edge, Firefox, Safari và nhiều mobile browser; có thể bị tắt do driver/GPU blacklist hoặc chính sách thiết bị. | Phù hợp `@tensorflow-models/*` hoặc face-api.js; không phải lựa chọn chính cho ONNX model PoC nếu WASM đã ổn. |
| WebNN | Browser gọi API WebNN để delegate toán ML xuống CPU/GPU/NPU tùy thiết bị. | Hạn chế nhất: phụ thuộc browser flag/phiên bản và phần cứng có NPU/accelerator. Không nên dùng làm baseline production. | Phù hợp thử nghiệm trên thiết bị có NPU/accelerator; chưa nên làm runtime chính cho MVP máy khách. |

Với MVP Gym offline-first, lựa chọn dễ kiểm soát nhất là giữ `WASM` làm baseline vì chạy được rộng và không phụ thuộc GPU. Sau đó mới benchmark thêm WebGPU/WebGL trên đúng máy khách nếu cần giảm latency.

### So sánh các model local và face-api.js

Trong phương án tự xử lý, có hai hướng triển khai thực tế: ghép các model tối ưu cho từng bước hoặc dùng face-api.js như một pipeline JavaScript thống nhất.

| Tiêu chí | MediaPipe + alignment + FaceNet/MobileFaceNet/ArcFace | face-api.js |
| --- | --- | --- |
| Cấu trúc pipeline | 4 bước rõ: BlazeFace detect -> align `112 x 112` -> ONNX embedding -> cosine Top K. | 3 bước cùng thư viện: face-api detector -> landmark optional -> FaceRecognitionNet descriptor. |
| Detector | Có model cụ thể trong repo: short-range `229.746 bytes`, full-range `1.083.786 bytes`; output có keypoint mắt/mũi. | TinyFaceDetector output `box + score`; muốn landmark/descriptor phải chạy thêm model khác. |
| Alignment | PoC đã có `THREE_POINT_BLAZEFACE` và `BOUNDING_BOX_FALLBACK`; có thể nâng lên 5 điểm khi thêm landmarker. | Cần thêm face-api landmark model; nếu bỏ alignment thì descriptor dễ dao động theo crop. |
| Embedding | PoC ONNX `13.616.099 bytes`, input `[1,3,112,112]`, config `threshold 0.62`, `margin 0.08`. | Descriptor của FaceRecognitionNet không dùng chung được với vector ONNX; phải tách `model_key`. |
| Accuracy cần benchmark | Benchmark bằng ảnh Gym thật: pass/fail theo `best_score`, `second_score`, `margin`; dễ thay model nếu kết quả kém. | Dùng làm baseline: so cùng tập ảnh, cùng enrollment, cùng threshold riêng; không giả định threshold giống ONNX. |
| Hiệu suất realtime | Detect liên tục, embedding chỉ chạy trên frame tốt nhất; mục tiêu UX là một lượt check-in dưới `1-2 giây`. | Nếu chạy detect + landmark + descriptor liên tục trên stream, dễ nặng; nên chỉ descriptor trên frame đã chọn. |
| Offline-first | Cache model ONNX ~`13 MB` + vector `~2 KB/vector 512D` + snapshot member trong IndexedDB. | Offline được nếu cache đủ weights, nhưng dung lượng và versioning của nhiều weight cần quản lý riêng. |
| Khả năng nâng cấp | Thay detector/alignment/embedding độc lập; đổi embedding thì re-embed toàn bộ vector theo `model_key`. | Dễ bắt đầu nhưng khó tối ưu từng phần; nâng model thường kéo theo đổi cả pipeline face-api.js. |

Kết luận: với máy khách tại Gym, hướng combo `MediaPipe + alignment rõ ràng + MobileFaceNet/ArcFace mobile` đáng ưu tiên hơn vì kiểm soát tốt từng tầng và dễ tối ưu theo thiết bị thật. `face-api.js` vẫn hữu ích để dựng PoC nhanh hoặc làm baseline, nhưng cần benchmark kỹ trước khi dùng cho check-in tự động.

## Phương án 2: Dùng dịch vụ đám mây

Phương án này gửi ảnh lên dịch vụ như AWS Rekognition hoặc Azure Face để so sánh, tìm người hoặc xác thực. Khi cần tìm kiếm 1:N, cloud thường yêu cầu đăng ký khuôn mặt vào collection, user/person hoặc PersonGroup do nhà cung cấp quản lý; hệ thống nhận lại `faceId`, candidate, similarity hoặc confidence, không nhận raw embedding để tự so khớp.

| Hạng mục | Tham chiếu cụ thể |
| --- | --- |
| Hạ tầng model/vector search | Cloud vận hành model và collection; hệ thống không nhận raw embedding để tự search offline. |
| Latency | Có upload ảnh + API response; đặt timeout vận hành ví dụ `2-3 giây`, quá ngưỡng thì fallback. |
| Chi phí request | Tính tuyến tính theo lượt gọi; ví dụ `100.000` request/tháng khoảng `100 USD` theo giả định giá bên dưới. |
| Dữ liệu trả về | Nhận `faceId`/candidate/similarity/quality/pose tùy API; không nhận vector nội bộ để cache IndexedDB. |
| Quyền riêng tư | Ảnh khuôn mặt rời khỏi máy khách/hệ thống nội bộ; cần consent, retention, audit access và quy trình xóa dữ liệu. |

### Dịch vụ đại diện

| Dịch vụ | Năng lực liên quan | Ghi chú nghiên cứu thực dụng |
| --- | --- | --- |
| AWS Rekognition | CompareFaces, collection search, trả similarity/candidate. | Hợp với benchmark online hoặc verify ảnh tĩnh; nếu dùng mỗi lượt check-in thì tính theo request, ví dụ `100.000` lượt/tháng ~`100 USD` theo giả định bên dưới. |
| Azure Face | Detect, Verify/Identify với PersonGroup/LargePersonGroup, trả thuộc tính/kết quả so khớp. | Cần kiểm tra quyền truy cập nhận diện khuôn mặt và region; không dùng được làm offline vector source vì không trả embedding để client tự so. |

### Độ khả dụng

Với Gym tại quầy, camera thường chạy như một luồng video liên tục. Cloud face recognition không nên nhận từng frame realtime từ stream vì chi phí, latency và băng thông sẽ tăng rất nhanh. Cách phù hợp hơn là client hoặc máy khách detect mặt trước, chọn frame tốt nhất, rồi chỉ gửi ảnh tĩnh lên cloud khi thật sự cần xác thực.

| Tiêu chí | AWS Rekognition / Azure Face | Nhận xét cho Gym |
| --- | --- | --- |
| Ảnh tĩnh sau khi chọn best frame | Gửi `1` ảnh cho mỗi lượt verify/search. | Client detect/quality local trước, chỉ gửi frame rõ nhất lên cloud. |
| Phụ thuộc mạng | Bắt buộc có upload ảnh và response API. | Đặt timeout, ví dụ `2-3 giây`; quá timeout thì chuyển offline/local hoặc QR để quầy không đứng. |
| Khả năng offline | Không đáp ứng. | Mất mạng thì không gọi được API; cần local model/vector hoặc fallback mã hội viên/số điện thoại. |
| Độ sẵn sàng vận hành | Phụ thuộc region, quota, rate limit và chính sách quyền truy cập. | Cần log `cloud_request_id`, latency, error code, retry count để biết lỗi do mạng, quota hay payload. |

## Phương án 3: Kết hợp trực tuyến và cục bộ

Phương án hybrid dùng xử lý cục bộ làm đường chính cho máy khách, sau đó dùng cloud khi online để kiểm tra lại, benchmark hoặc xử lý ca nhập nhằng.

| Tiêu chí | Mốc triển khai hybrid | Chỉ số cần đo |
| --- | --- | --- |
| Accuracy | Local quyết định ca rõ; cloud/server chỉ kiểm tra ca `AMBIGUOUS`, ví dụ best-score đạt threshold nhưng margin `< 0.08`. | Tỷ lệ false accept/false reject theo tập ảnh Gym thật; tỷ lệ ca `AMBIGUOUS`. |
| Speed | Luồng thường chạy local; chỉ ca khó mới gọi online với timeout ví dụ `2-3 giây`. | P50/P95 latency cho local-only và cloud-fallback. |
| Offline | Client cần đủ model ~`13 MB`, vector ~`2 KB/vector`, snapshot member, threshold/margin trong IndexedDB. | Số ngày snapshot còn chấp nhận, số item queue pending, tỷ lệ sync thành công. |
| Cost | Nếu chỉ gửi `5-10%` ca khó lên cloud, chi phí request giảm mạnh so với gửi `100%` lượt check-in. | Tỷ lệ request cloud thực tế trên tổng lượt check-in. |
| Privacy | Ảnh/vector phần lớn ở nội bộ; chỉ ảnh ca khó mới rời hệ thống. | Số ảnh gửi cloud/ngày, retention, audit access, yêu cầu xóa dữ liệu. |

Luồng offline trong phương án hybrid cần được thiết kế rõ:

1. Khi máy khách online, client sync snapshot gồm member hợp lệ, vector, `model_key`, `threshold`, `margin`, `top_k` và `snapshot_version` xuống IndexedDB.
2. Khi check-in, client có thể thử request online trước để server verify hoặc tạo check-in ngay.
3. Nếu request lỗi tạm thời như mất mạng, timeout, `502`, `503` hoặc `504`, client fallback sang offline recognition: tạo embedding local, so với vector đã cache, kiểm tra threshold/margin/cooldown.
4. Nếu offline recognition `PASSED`, client tạo check-in local với `client_event_id`, `idempotency_key`, `recognition_source = "offline_client"` và đưa vào queue IndexedDB.
5. Nếu kết quả `FAILED` hoặc `AMBIGUOUS`, không queue face check-in tự động; chuyển sang QR, mã hội viên hoặc số điện thoại.
6. Khi online lại, client gửi batch queue lên server; server xử lý từng item theo `client_event_id`/`idempotency_key` và trả `stored`, `duplicate`, `rejected` hoặc `conflict`.
7. Client xóa item `stored`/`duplicate`, giữ item `rejected`/`conflict` để audit, sau đó refresh snapshot member/vector mới nhất.

Điểm mấu chốt là cloud hoặc server online không thay thế snapshot offline. Máy khách muốn tự đứng được khi mất mạng thì luôn phải có model local, member/vector local, queue local và cơ chế sync idempotent.

## Ước tính chi phí

Với AWS Rekognition, nhóm API liên quan đến so sánh/tìm kiếm khuôn mặt như `CompareFaces`, `IndexFaces`, `SearchFacesByImage`, `SearchFaces` thuộc Group 1. Trang giá AWS nêu mức **0.001 USD cho mỗi ảnh** ở một triệu ảnh đầu tiên của nhóm này. Nếu dùng collection để tìm kiếm khuôn mặt, AWS còn tính phí lưu face/user vector theo tháng.

Với Azure Face, gói Standard Web tính theo mỗi 1.000 transaction. Azure pricing có free tier **30.000 transaction/tháng** ở một số vùng. Ví dụ dưới đây dùng giả định giá Standard Web là **1 USD / 1.000 transaction** theo Azure Retail Prices tại thời điểm tra cứu; khi lập dự toán thật nên chốt lại bằng Azure Pricing Calculator theo region và offer đang dùng.

| Quy mô request/tháng | AWS Rekognition Group 1 | Azure Face Standard Web |
| --- | --- | --- |
| 100.000 request | `100.000 x 0.001 USD` = khoảng `100 USD/tháng` | `100.000 / 1.000 x 1 USD` = khoảng `100 USD/tháng`; nếu trừ free tier 30.000 thì còn khoảng `70 USD/tháng`. |
| 1.000.000 request | `1.000.000 x 0.001 USD` = khoảng `1.000 USD/tháng` | `1.000.000 / 1.000 x 1 USD` = khoảng `1.000 USD/tháng`; nếu trừ free tier 30.000 thì còn khoảng `970 USD/tháng`. |

Các con số trên chỉ tính request nhận diện/xác thực cơ bản. Chưa tính lưu trữ face vector, Face Liveness, truyền dữ liệu, lưu ảnh, thuế, chiết khấu cam kết hoặc khác biệt giá theo vùng.

## Dữ liệu cần lưu để cải thiện dần

| Trường | Ví dụ giá trị | Dùng để quyết định/điều tra gì |
| --- | --- | --- |
| `matched_member_id` | `member_123` | Hội viên được nhận diện hoặc `null` nếu failed/ambiguous. |
| `similarity_score` | `0.84` | So với `threshold`; nếu thấp hơn ngưỡng thì không auto check-in. |
| `second_score` | `0.77` | Tính margin; ví dụ `0.84 - 0.77 = 0.07` nhỏ hơn `0.08` thì ambiguous. |
| `quality_score` | `0.72` hoặc blur/brightness riêng | Phân biệt model sai do mặt giống nhau hay ảnh đầu vào xấu. |
| `model_key` | `buffalo_sc_w600k_mbf` | Biết vector nào tương thích; đổi model thì phải re-embed. |
| `threshold` | `0.62` | Ngưỡng đang áp dụng tại thời điểm quyết định, không ghi đè bằng config mới sau này. |
| `margin` | `0.08` | Khoảng cách yêu cầu giữa best và second candidate. |
| `alignment_method` | `THREE_POINT_BLAZEFACE` | So sánh lỗi giữa 5-point, 3-point và bounding-box fallback. |
| `recognition_source` | `offline_client`, `server`, `cloud` | Biết quyết định đến từ client offline, server nội bộ hay cloud. |
| `fallback_reason` | `best_below_threshold`, `margin_too_small`, `network_timeout` | Thống kê vì sao user phải dùng QR/số điện thoại. |

Các dữ liệu này giúp nhóm phát triển tinh chỉnh ngưỡng, so sánh model và đánh giá camera thực tế tại từng cơ sở.
